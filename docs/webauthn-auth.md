# WebAuthn / Passkey Authentication

Today Voxply generates an Ed25519 keypair from a random seed, stores
the seed in `localStorage`, and uses session tokens derived from that
identity to authenticate with hubs. This works but has two pain points:

- Users must back up a recovery phrase to move between devices.
- The private key lives in `localStorage`, which survives neither a
  hard refresh after `localStorage.clear()` nor a hub restart that
  generates a new identity (invalidating all stored sessions).

**WebAuthn** (Web Authentication API, FIDO2) replaces both. The
user's keypair lives in the device's secure enclave (TPM, Secure
Enclave, Android StrongBox) or on a hardware security key (YubiKey).
Authentication is a biometric/PIN tap — no phrase, no clipboard.

---

## What changes and what doesn't

| Today | After WebAuthn |
|---|---|
| Seed generated in-browser, stored in `localStorage` | Keypair generated by authenticator, stored in secure hardware |
| Recovery phrase backs up the seed | Cross-device sync via passkey provider (iCloud Keychain, Google PM, 1Password) or hardware key |
| Hub restart with new identity invalidates all sessions | Sessions are re-established per hub, independent of hub identity rotation |
| Private key accessible to JS (extraction risk) | Private key never leaves the authenticator |
| Works on HTTP | Registration/assertion require HTTPS except on `localhost` |

Identity is **still per-user keypair** — WebAuthn is just a better
storage and ceremony for it. The Ed25519 identity key (used for
message signing, certifications, DMs) stays; only the
**authentication ceremony** with each hub changes.

## Protocol sketch

### Registration (first join on a device)

```
Browser                         Hub
  |                               |
  |-- POST /auth/webauthn/begin ->|
  |                               | generate PublicKeyCredentialCreationOptions
  |                               | (challenge, rpId, user.id = pubkey)
  |<- { options } ---------------|
  |                               |
  | navigator.credentials.create(options)
  | [user taps biometric/PIN]     |
  | authenticator creates keypair |
  | returns PublicKeyCredentialResponse
  |                               |
  |-- POST /auth/webauthn/finish->|
  |   { credential }              | verify attestation
  |                               | store (user_id, credential_id, public_key, aaguid)
  |<- { session_token } ---------|
```

### Login (returning user)

```
Browser                         Hub
  |                               |
  |-- POST /auth/webauthn/assert/begin ->|
  |                               | generate PublicKeyCredentialRequestOptions
  |                               | (challenge, allowCredentials for this user)
  |<- { options } ---------------|
  |                               |
  | navigator.credentials.get(options)
  | [user taps biometric/PIN]     |
  | authenticator signs challenge |
  | returns AuthenticatorAssertionResponse
  |                               |
  |-- POST /auth/webauthn/assert/finish ->|
  |   { assertion }               | verify signature against stored pubkey
  |                               | issue session_token
  |<- { session_token } ---------|
```

Session tokens are the same bearer tokens Voxply already uses
everywhere — no downstream changes to WS auth, `hubFetch`, or route
guards.

## Hub changes

### New dependency

```toml
# hub/Cargo.toml
webauthn-rs = { version = "0.5", features = ["danger-allow-state-serialisation"] }
```

`webauthn-rs` handles challenge generation, CBOR credential parsing,
attestation verification, and assertion validation. It is maintained
by Firstyear (William Brown) and is the de-facto standard Rust crate
for this.

### New DB table

```sql
CREATE TABLE webauthn_credentials (
    id             TEXT PRIMARY KEY,   -- base64url credential_id
    user_id        TEXT NOT NULL,      -- FK → users.public_key
    public_key_cbor BLOB NOT NULL,     -- COSEKey from registration
    sign_count     INTEGER NOT NULL DEFAULT 0,
    aaguid         TEXT,               -- authenticator model hint
    created_at     INTEGER NOT NULL,
    last_used_at   INTEGER
);
```

One user can have multiple credentials (phone + laptop + hardware
key). All are valid; the hub verifies whichever credential signed
the assertion.

### New routes

| Method | Path | Purpose |
|---|---|---|
| POST | `/auth/webauthn/begin` | Start registration; returns `PublicKeyCredentialCreationOptions` |
| POST | `/auth/webauthn/finish` | Complete registration; stores credential, issues session token |
| POST | `/auth/webauthn/assert/begin` | Start assertion (login); returns `PublicKeyCredentialRequestOptions` |
| POST | `/auth/webauthn/assert/finish` | Complete assertion; verifies, issues session token |
| GET | `/me/credentials` | List registered passkeys (name, aaguid, created_at, last_used_at) |
| DELETE | `/me/credentials/:id` | Remove a passkey (requires at least one remaining, or an active session) |

The existing `/auth/register` + `/auth/login` keypair flow stays in
place for desktop/android clients that don't have a browser
authenticator. WebAuthn is additive.

### State

`AppState` gains a `Webauthn` instance (from `webauthn-rs::WebauthnBuilder`),
configured with:

```rust
WebauthnBuilder::new(rp_id, rp_origin)?
    .rp_name("Voxply")
    .build()?
```

`rp_id` is the hub's public domain (e.g. `voxply.videogamezone.eu`).
For `localhost` dev it is `"localhost"`.

In-flight challenges (begin → finish window, ~5 min TTL) live in a
`DashMap<session_id, RegistrationState>` / `DashMap<session_id,
AuthenticationState>` in `AppState`, not in the DB. They are
passivated to disk only if `danger-allow-state-serialisation` is needed
across hub restarts (optional).

## Client changes (web)

The web client handles both flows. Desktop/android use the
existing keypair flow and are unaffected.

### New platform function (`apps/web/src/platform/webauthn.ts`)

```ts
export async function startRegistration(options: PublicKeyCredentialCreationOptionsJSON): Promise<RegistrationResponseJSON>
export async function startAssertion(options: PublicKeyCredentialRequestOptionsJSON): Promise<AuthenticationResponseJSON>
```

These are thin wrappers around `navigator.credentials.create` /
`navigator.credentials.get` that handle base64url encoding. The
`@simplewebauthn/browser` package (3 kB gzipped) provides
battle-tested wrappers already — no need to roll them.

```
pnpm --filter voxply-web add @simplewebauthn/browser
```

### Auth flow in `WelcomeScreen` / `AddHubModal`

Replace the current "Create identity → copy seed → join hub" linear
flow with:

1. User enters hub URL.
2. Client fetches `GET /hub-info` (public, no auth).
3. If hub supports WebAuthn (`hub_info.webauthn_enabled: true`):
   - Show "Sign in with passkey" button.
   - On click: call `POST /auth/webauthn/assert/begin`, drive
     `navigator.credentials.get`, POST result to
     `/auth/webauthn/assert/finish`, receive session token.
   - First-time only: call `POST /auth/webauthn/begin`, drive
     `navigator.credentials.create`, POST result to
     `/auth/webauthn/finish`.
4. Store only the session token and hub URL in `localStorage`
   (no seed, no keypair material). The session token is short-lived
   and scoped to one hub — its loss is not identity loss.

### Identity key

The user's Ed25519 identity key (used for message signing and DMs)
still needs to live somewhere. Options, in order of preference:

| Option | Pro | Con |
|---|---|---|
| Derive from WebAuthn PRF extension | Key never touches JS memory | PRF not widely supported yet (Chrome 116+, Safari 17+) |
| Generate once, store in `localStorage`, tie to passkey with a key-wrapping step | Solid today | Still uses `localStorage` for the wrapped key |
| Hub stores per-user identity key (operator trust model) | Zero client storage | Operator can impersonate users |

**Recommended for v1**: generate the Ed25519 key once on first
registration, wrap it with a key derived from the WebAuthn PRF
extension (or a random AES key stored in `localStorage` as a
fallback). The session token authenticates you to the hub; the
wrapped key authenticates your *messages*. This splits concerns
cleanly and can be upgraded to pure-PRF when browser support is
universal.

## Migration path

Existing users (seed-phrase identity) keep working — the current
`/auth/register` flow is not removed. They can **add** a passkey at
`Settings → Account → Add passkey` which calls
`POST /auth/webauthn/begin` on their current authenticated session,
linking the credential to their existing `public_key`.

---

## HTTPS requirement

WebAuthn is blocked on plain HTTP except on `localhost`. The pilot
hub (`voxply.videogamezone.eu`) already terminates TLS at Cloudflare,
so it works. Any new hub that wants passkeys needs TLS — document
this in `hosting.md`.

## Open questions

- Should `rp_id` be configurable per hub (for operators who run
  behind a reverse proxy with a different hostname), or always
  derived from `VOXPLY_PUBLIC_URL`?
- Should the existing seed-phrase flow be deprecated on a schedule
  once WebAuthn adoption is high enough to justify removing it?
- Do we want to support **conditional UI** (autofill-driven passkey
  prompts) for the hub URL input field?
