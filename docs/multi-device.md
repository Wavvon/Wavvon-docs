# Multi-Device Pairing

**Status**: design — no code yet. This doc covers the
identity model and the QR pairing flow. The storage layer that holds
the device registry, revocation list, and prefs blob lives in
[home-hub.md](home-hub.md) — read that first if you haven't.

The goal: one user, many devices (phone + desktop + laptop), under a
single identity. Today every device generates its own keypair and is
treated as a separate user; pasting the recovery phrase on a second
device *replaces* the second device's identity instead of pairing it.

## Decisions (TL;DR)

| Question | Choice |
|---|---|
| Identity model | **Master + per-device subkeys.** Master is derived from the existing 24-word BIP39 phrase. Each device gets its own subkey, signed by master. |
| Storage | **Home hub list** — see [home-hub.md](home-hub.md). |
| Pairing UX | **QR code.** Existing device shows a code; new device scans. |
| Existing identities | **Migrate as `subkey 0`.** The current single-key identity becomes the master's first authorized subkey, no on-the-wire change for hubs that haven't upgraded. |

## Identity model — master + per-device subkeys

### Derivation

The 24-word BIP39 phrase is treated as an HD-style seed. From it we
derive:

- **Master keypair** (Ed25519): never used to sign daily traffic. Only
  signs subkey certs, revocations, and `HomeHubList` designations
  ([home-hub.md](home-hub.md)). Held by any device that has the phrase
  but kept cold — unlocked only when enrolling, revoking, or moving
  the home hub list.
- **Subkey 0** (Ed25519): the existing single-key identity, kept as
  the first device's working key. This is what makes existing users
  migrate cleanly — their current key *is* subkey 0.

Derivation: `HKDF-SHA256(salt=∅, ikm=phrase_entropy,
info="wavvon/master/v1", L=32)` for master. **Subkey 0 is the existing
per-device key** — the BIP39 entropy bytes used directly as an Ed25519
secret, preserving today's derivation so non-upgraded hubs see no
change in pubkey. Subkeys 1+ are generated by random keygen on the new
device (not derived from the phrase), so a device that never sees the
phrase still holds a real secret.

Code home: `identity/src/lib.rs` in Wavvon-server adds `MasterIdentity`
and `DeviceSubkey` next to existing `Identity`.

### Subkey cert

When a device is enrolled, the master signs a cert:

```
SubkeyCert {
  master_pubkey:   Ed25519PublicKey,
  subkey_pubkey:   Ed25519PublicKey,
  device_label:    String,            // user-set, e.g. "Phone"
  issued_at:       u64,               // unix seconds
  not_after:       Option<u64>,       // optional expiry
  fallback_hubs:   Vec<String>,       // designation-of-last-resort hubs
  signature:       Ed25519Signature,  // master sig over the bytes above
}
```

The cert is what a hub verifies during auth: subkey signs the
challenge nonce *and* presents its cert; hub verifies both signatures
and that `master_pubkey` is the user's identity. The `fallback_hubs`
field carries the "designation-of-last-resort" list described in
[home-hub.md](home-hub.md) — used only when the active home hub list
is fully unreachable.

### Revocation

The master signs a `RevocationEntry { master_pubkey, subkey_pubkey,
revoked_at, signature }`. Entries are append-only, published by every
home hub in the user's list. Other hubs cache them with a short TTL
(e.g. 1 hour) and refuse auth from any subkey listed.

A revoked subkey can never be un-revoked (an attacker who saw it once
must remain locked out). To re-add a device, generate a new subkey.

## QR pairing protocol — message by message

Existing device = **E** (already enrolled, holds master). New device =
**N** (fresh install, no identity).

### Step 1 — N declares itself

N generates a fresh subkey pair `(N_pub, N_priv)`. UI: "Pair this
device — show this device's screen to a paired device."

### Step 2 — E generates a pairing offer

User taps "Pair a device" on E. E:

1. Generates a one-time pairing token `T` (32 bytes random).
2. Composes a QR payload:
   ```
   PairingOffer {
     master_pubkey:    Ed25519PublicKey,
     home_hub_list:    Vec<String>,        // full list from current designation
     pairing_token:    [u8; 32],
     issued_at:        u64,
     expires_at:       u64,                // issued_at + 5 minutes
   }
   ```
3. POSTs `T` + `expires_at` to **every reachable home hub** in the
   list, at `/identity/pairing/offer`, authenticated as E's subkey.
   Each home hub stores the offer in a short-lived table keyed by
   token. Posting to all of them ensures N can claim against any
   reachable home hub.
4. Renders the QR code from the JSON payload.

### Step 3 — N scans

N decodes the QR. It now knows: master pubkey, home hub list,
pairing token. N tries each URL in `home_hub_list` in order until
one responds, then opens a WebSocket to that hub at
`/identity/pairing/claim`:

```
ClaimRequest {
  pairing_token: [u8; 32],
  subkey_pubkey: Ed25519PublicKey,    // N_pub
  device_label:  String,              // user-typed, e.g. "iPhone"
  proof:         Ed25519Signature,    // N_priv signs the token
}
```

The home hub checks the token is live, holds the request, and pushes
a notification to E over E's existing WebSocket session on any hub
where E is online.

### Step 4 — E confirms

E receives "device 'iPhone' wants to pair, fingerprint `ab:cd:..:ef`."
User taps confirm. E:

1. Constructs a `SubkeyCert` for N_pub with the chosen label.
2. Signs the cert with master.
3. Constructs a `BlobKeyShare`: the prefs-blob symmetric key wrapped
   for N_pub (X25519 ECIES — a one-shot DH using N's pubkey converted
   to its X25519 form, plus an authenticated AEAD).
4. POSTs `{ pairing_token, cert, blob_key_share }` to the *same home
   hub* N opened the WebSocket against, at
   `/identity/pairing/complete`, authenticated as E.

### Step 5 — Home hub finalizes

The home hub:

1. Verifies cert signature against master pubkey.
2. Appends cert to the device registry.
3. Forwards `{ cert, blob_key_share }` over the WebSocket N is
   holding open.
4. Deletes the pairing token from its own store.
5. Replicates the cert to peer home hubs in the list (best-effort,
   with retry — eventual consistency per [home-hub.md](home-hub.md)).

### Step 6 — N is paired

N stores cert + subkey + blob key. Fetches the encrypted prefs blob
from any hub in the home hub list, decrypts, populates local hub
list / friends / prefs. From this point N authenticates to any hub
by presenting `{ subkey_sig, cert }` — no further interaction with
home hubs needed for daily use beyond reading DMs and pushing prefs
updates.

### Security properties

- Pairing token is one-shot, 5-minute TTL, bound to N's pubkey via
  the proof signature.
- Master is never sent over the wire. E performs the signing locally.
- Blob symmetric key is wrapped per-device, never stored on home
  hubs in plaintext.
- An attacker who intercepts the QR offer and races N still needs E's
  user to confirm a stranger's fingerprint — confirmation UI shows
  the fingerprint of the claiming subkey.
- Posting offers to multiple home hubs means a single hostile home
  hub cannot block pairing — N tries other hubs in the list.

## Hub-side changes (auth)

What every hub (not just home hubs) must implement to participate:

- **Auth path** accepts `{ subkey_sig, subkey_cert }` instead of bare
  `subkey_sig`. Verify cert signature, check `master_pubkey` is the
  identity, check cert not expired. Cache the subkey→master mapping.
- **Revocation cache**: periodically fetch revocation entries from
  any hub in the user's home hub list (per
  [home-hub.md](home-hub.md)). Refuse auth from listed subkeys.
- **Federation auth**: hub-to-hub auth is unchanged. User-on-behalf
  federation messages now carry the user's master pubkey, not
  subkey, as the canonical identifier.

The home-hub-only endpoints (`/identity/pairing/*`, device registry,
revocations) are documented in [home-hub.md](home-hub.md). Pairing
endpoints are short-lived state and are the only home-hub endpoints
that interact directly with this pairing protocol.

## Migration of existing single-key identities

The current `Identity` (single Ed25519 key derived from the phrase)
becomes `subkey 0` of a master derived from the same phrase. On
upgrade:

1. Client computes master from existing phrase.
2. Master self-signs a `SubkeyCert` for the existing key, labeled
   "this device" by default.
3. Client prompts user to pick a home hub list, pre-filled with the
   first 1-2 hubs in their hub list.
4. Client posts the cert and the initial `HomeHubList` designation
   to every hub in the list.
5. Future logins to any upgraded hub use `{ existing_key_sig, cert }`.
   Logins to a hub that hasn't upgraded yet still work because the
   subkey *is* the existing pubkey — old hubs see the same identity
   they always saw.

This is the property that makes the migration painless: subkey 0 has
the same pubkey as today's identity, so non-upgraded hubs see no
change.

## Threat model deltas (pairing-specific)

For the broader home-hub threat model see [home-hub.md](home-hub.md).
Pairing-specific surfaces:

| Surface | Mitigation |
|---|---|
| Race during pairing (attacker scans QR before legit device) | Confirm UI on E shows the fingerprint of the claiming subkey. User must confirm a stranger's fingerprint to pair. |
| Hostile home hub blocks pairing | Offer is posted to all hubs in the list; N tries each. |
| Compromised subkey | Master revokes via a `RevocationEntry`, propagated through the home hub list. |
| Compromised master (phrase leaked) | Same as today — phrase compromise = identity compromise. Revocation can't save you if master itself is leaked. New phrase = new identity. |

## Phasing

1. **Phase 1 — derive master + subkey 0, no protocol change.** Add
   master derivation, mint subkey 0 from existing phrase, wire
   `MasterIdentity` types. Ship as a no-op refactor; existing users
   see nothing.
2. **Phase 2 — home hub list endpoints.** Implement device registry,
   revocations, prefs blob, designation storage on every hub
   ([home-hub.md](home-hub.md)). Hubs serve them but no client uses
   them yet.
3. **Phase 3 — home hub picker + pairing UI.** First-run picker for
   the home hub list, QR generation, scan flow, confirm UI, blob
   sync.
4. **Phase 4 — auth uses subkey certs.** Hubs accept `(sig, cert)`
   tuples. Revocation cache active. Old single-key auth still
   accepted for unupgraded clients.
5. **Phase 5 — list-management UX + DMs through home hubs.** Add/
   remove/reorder home hubs, divergence alerts, DM delivery
   reorganization to use the home hub list.

## What's deferred (explicitly)

- **Cross-device push for chat.** Multi-device means a notification
  could fire on phone *and* desktop. Read-receipt-style suppression
  is its own design.
- **Per-device permissions.** All subkeys are equally privileged
  today. A future "this subkey is read-only / phone-restricted"
  capability is layerable on the cert.
- **Sync of voice settings between devices.** The prefs blob includes
  voice settings, but per-device overrides (different mic on
  different devices) need a UI distinction.

## Implementation — DM attribution & DH fix

Fixes the bug where a paired device attributes DMs to its subkey and
keys E2E against the subkey's X25519 key instead of the canonical
identity's. Rationale and alternatives are in
[decisions.md](decisions.md). Community-axis actions already resolve to
canonical via `resolve_canonical_identity`; this brings the DM path in
line without ever putting a signing seed on a paired device.

The delivery target today is the web client, so the client work below is
in Wavvon-clients (`packages/core`, `apps/web`); the desktop/Android
Tauri equivalents follow the same shape when those ship.

### Mechanism A — DH capability via a wrapped canonical scalar

The canonical DM DH keypair is unchanged: the X25519 scalar from the
canonical (subkey-0 / entropy) Ed25519 seed via SHA-512+clamp
(`dhKeypairFromSeed`), published at `/identity/{canonical}/dh-key`. A
paired device is provisioned with a copy of that scalar at pairing.

Server (Wavvon-server):

- `identity/src/lib.rs` — `PairingComplete` (and the wire vectors)
  gain an optional `wrapped_dh_seed_hex: Option<String>` alongside
  `wrapped_blob_key_hex`. Both are ECIES-wrapped for the new subkey; the
  identity crate already exposes the wrap/unwrap primitive.
- No DB or `dh_keys` change — the published DH key stays keyed to the
  canonical pubkey.

Client (Wavvon-clients):

- `packages/core/src/identity/wire.ts` — add `wrapped_dh_seed_hex?:
  string` to `PairingComplete` / `PairingStatus`.
- Enrolling device (`apps/web/.../DevicesSection.tsx`, step 4): wrap
  `x25519PrivFromSeed(entropy)` (the 32-byte clamped scalar, **not** the
  Ed25519 seed) for the claiming subkey with `wrapBlobKey` and include it
  in the complete payload.
- Claiming device (`apps/web/.../IdentitySetupScreen.tsx`): unwrap with
  `unwrapBlobKey` and persist the scalar on the account record
  (`identity/store.ts` `IdentityRecord.canonical_dh_priv_hex?`), in the
  account-scoped namespace.
- `packages/core/src/identity/crypto.ts` — let the 1:1 and DR static
  steps take the canonical DH **private scalar** directly rather than
  always deriving it from a seed: add an optional scalar parameter to
  `initDrSession` / `decryptDmDr` (and the v1 `encryptDm`/`decryptDm`
  DH inputs). The primary device passes the entropy-derived scalar (as
  today); a paired device passes the stored `canonical_dh_priv_hex`.
- `apps/web/src/platform/commands/dms.ts` — `sendDm`/`getDmMessages`
  select the DH scalar: `canonical_dh_priv_hex` if present, else
  `dhKeypairFromSeed(seed_hex)`. Recipient selection and `fetchDhKey`
  stay keyed to the canonical pubkey.
- `publishDhKey` — guard: only publish when the device holds the
  canonical signing seed (i.e. `publicKeyHex(seed_hex) === canonical`
  and no `signer_cert`). Paired devices skip publish; the primary already
  published the canonical DH key.

### Mechanism B — attribution via cert-chained envelopes

Envelope keeps `sender_pubkey = canonical`. A paired device signs with
its subkey and attaches its cert; primary/legacy devices send exactly as
today (no cert, byte-identical envelope — no wire-vector break).

Server (Wavvon-server):

- `hub/src/routes/dm_models.rs` — `EncryptedDmEnvelope` and
  `FederatedDmRequest` gain `signer_cert: Option<SubkeyCert>`.
- `hub/src/routes/dms/keys.rs` — `envelope_signing_bytes` is unchanged
  (`sender_pubkey` is not in the signed bytes). Add a verifier helper
  `verify_envelope_sender(env, expected_canonical, master_lookup)`:
  - `signer_cert` absent → verify signature against `sender_pubkey`
    (current behaviour).
  - present → verify cert (master→subkey), verify signature against
    `cert.subkey_pubkey`, and require `sender_pubkey` to bind to
    `cert.master_pubkey`.
- `hub/src/routes/dms/messages.rs`:
  - `send_dm` — replace the `verify_signature(&user.public_key, …)` call
    with the helper. Binding is trivial here: require `signer_cert
    .master_pubkey == user.master_pubkey` and `env.sender_pubkey ==
    user.public_key` (both already on the authenticated session; add
    `master_pubkey` to `AuthUser` — see auth note). Store/broadcast
    `sender = user.public_key` (canonical), and forward `signer_cert` in
    the outbox `FederatedDmRequest`.
  - `receive_federated_dm` — verify via the helper. Binding: resolve
    `master_pubkey → canonical` from the local `users` row; if the sender
    is unknown, fetch the sender's device registry from a delivery hub
    and require the same master to have certified `sender_pubkey`
    (canonical's self-cert), then cache. Store under the resolved
    canonical.
- `hub/src/auth/middleware.rs` / `auth/models.rs` — expose the session's
  `master_pubkey` on `AuthUser` (already resolved in
  `resolve_canonical_identity`) so `send_dm` can bind without a re-query.

Client (Wavvon-clients):

- `packages/core/src/identity/crypto.ts` — `DmEnvelope` / `DrEnvelope`
  gain `signer_cert?: SubkeyCert`. `encryptDm`/`encryptDmDr` take the
  signing seed (the device subkey) **and** the canonical `sender_pubkey`
  + optional `signer_cert`; set `sender_pubkey = canonical` always,
  attach `signer_cert` only when the signing key ≠ canonical.
- Receive/verify path (`getDmMessages`) — when `signer_cert` is present,
  verify the two-link chain before trusting attribution; bind
  `sender_pubkey` to the conversation's canonical member.
- `apps/web/src/platform/commands/dms.ts` — `sendDm` passes
  `identity.canonical_pubkey ?? publicKeyHex(seed_hex)` as
  `sender_pubkey` and `identity.subkey_cert` as `signer_cert`.

### Phasing

1. **Server accepts both forms.** Add the optional envelope/pairing
   fields and the verifier helper; primary-device and legacy DMs are
   unaffected. `AuthUser.master_pubkey` exposed.
2. **Pairing hands out the DH scalar.** `wrapped_dh_seed_hex` on
   complete; claiming device stores it. New pairings are fixed.
3. **Client DM path uses canonical anchor.** DH-scalar selection,
   cert-chained signing, publish guard, receive-side chain verification.
4. **Federated binding hardening.** Device-registry fallback + cache in
   `receive_federated_dm` for unknown senders.

### Compatibility

- Historical rows are not rewritten. Cert-less subkey-keyed encrypted
  rows are effectively absent (they failed the pre-fix canonical
  signature check); any orphaned ones stay unreadable — a bounded,
  documented loss confined to the recent web-only pairing window.
- A cert-chained envelope reaching an un-upgraded hub or client fails its
  signature check (unknown `signer_cert`, sig verified against canonical)
  and does not federate/decrypt until that peer upgrades. Strict
  improvement over today, where paired-device E2E did not work at all;
  the un-upgraded population shrinks as the web client updates.
