# Android Client

A third client living at `apps/android/android/` in the **Wavvon-client**
monorepo that wraps the **browser client's** React UI and platform layer
inside a Tauri 2 Android shell. Tauri 2 supports Android with no licensing
cost; the Android keystore is free. We use Tauri for the OS shell (window,
status bar, deep links, secure storage) and — unlike the original design —
for an on-device voice pipeline.

> Historical note: this doc was written when Android was its own
> `Wavvon-android` repo, shared code came in through cross-repo Vite
> aliases, and voice was deferred. All three changed: the clients
> consolidated into the Wavvon-client monorepo (shared code lives in
> `packages/*`; see [client-monorepo.md](client-monorepo.md)), and
> **voice shipped on Android (2026-06-13)** using the native `wavvon-voice`
> UDP pipeline driven through Tauri commands — Android speaks UDP voice
> like desktop, not the browser's WS relay. The cross-repo-alias prose
> below is kept as design history.

The Android client is now near feature-parity. Voice works (native UDP
pipeline); screen share is still deferred (no `getDisplayMedia` in the
Android WebView), and the layout assumes a portrait phone viewport.

## Architecture: browser platform layer, not invoke()

Tauri 2 on Android *does* run a Rust library, but it runs it through
JNI inside the WebView's process — there is no second OS process, no
filesystem like `~/.wavvon`, and many `std::net` calls behave
differently than on desktop. Two options:

| Option | Verdict |
|---|---|
| Port the desktop Rust backend (`desktop/src-tauri/src/lib.rs` in Wavvon-desktop) to Android via Tauri's mobile plugin model — keep the `invoke()` surface | Maintenance: two Rust backends to keep in sync (Tauri commands' signatures, the WS task, the voice pipeline). Rejected. |
| **Reuse Wavvon-web's `@platform` layer verbatim**: HTTP via `fetch`, WS via `WebSocket`, identity in encrypted storage. Tauri is just the shell. | Pick this. The browser client already proved the surface; Android inherits it for free. |

The Android client imports the **same** `web/src/platform/` modules
from Wavvon-web through a Vite alias (the repos are siblings on disk,
or pulled in via npm workspace / git submodule depending on the
chosen sharing mechanism — see the open question below). No separate
adapter. The only Android-specific code is (a) identity storage
(secure keystore-backed instead of IndexedDB), (b) a thin Tauri
command set for OS integration (notifications, deep links, "share"
intent), and (c) any UI adjustments.

### Project layout

Inside the Wavvon-android repo:

```
android/
├── package.json
├── vite.config.ts            aliases to shared component + platform sources
│                             (from Wavvon-desktop and Wavvon-web)
├── index.html
├── src/
│   ├── main.tsx              entry, same shape as web
│   ├── App.tsx               shared via alias with desktop
│   ├── platform-android/
│   │   ├── identity-store.ts secure-storage wrapper (see below)
│   │   └── os.ts             notifications, share, deep-link bridge
│   └── styles-mobile.css     mobile-specific overrides on top of styles.css
└── src-tauri/
    ├── Cargo.toml            minimal: tauri + tauri-plugin-* for android
    ├── tauri.conf.json       android target, identifier com.wavvon.android
    ├── gen/android/          generated by `tauri android init`
    └── src/lib.rs            ~50 lines: setup, plugin registration, no commands beyond OS glue
```

The Rust side carries no networking business logic — no hub HTTP, no WS,
no crypto. The hub/DM/identity surface that lives in
`apps/desktop/src-tauri/src/lib.rs` (Wavvon-client) has a TypeScript twin
in the shared platform layer (`packages/platform`), and that twin is what
Android uses. The one exception added later is **voice**: Android drives
the native `wavvon-voice` UDP pipeline through a set of ~8 Tauri commands
(join/leave, mute, deafen, device select, etc.), the same crate the
desktop uses (see [voice.md](voice.md)).

## UI strategy: shared components, mobile overlay CSS

The desktop layout (HubSidebar | ChannelSidebar | ContentArea — see
[client.md](client.md)) does not fit a phone. Three options:

| Option | Verdict |
|---|---|
| Separate Android-only React tree | Drift; rejected |
| Responsive CSS only, same components | The components carry desktop-specific keyboard handlers (Ctrl+K switcher), hover menus, drag-drop reorder — touch ports are non-trivial |
| **Shared components with a mobile shell wrapper + CSS overlay**: at narrow viewports `App.tsx` renders a drawer-based navigation; the underlying components (`MessageList`, `MessageInput`, `MemberList`) are reused as-is | Pick this |

**Concretely**:

- A new `MobileShell` component (lives in
  `desktop/src/components/MobileShell.tsx` in Wavvon-desktop so all
  three clients can use it via the alias) wraps the three-pane layout in a drawer
  pattern: tap a hamburger to slide in the hub list, tap again for the
  channel list, swipe right to reveal members.
- The breakpoint is `max-width: 768px`. Below that, `MobileShell`
  takes over; above it, the existing layout renders. The Android client
  forces the mobile breakpoint regardless of viewport math (a
  user-agent guard) so a tablet-sized Android device still gets the
  drawer UX.
- Bottom-nav-bar is rejected — Wavvon has two axes (hub list + channel
  list) and bottom-nav assumes one. Drawers express both.
- Touch targets: lift hit areas to 44 px minimum via the overlay CSS;
  components don't change.
- Virtual keyboard: use the `visualViewport` API to resize the message
  composer area when the keyboard opens. No native Android plugin
  needed.
- Back-button: Tauri 2 emits a back-press event on Android; the shell
  handler closes any open drawer/modal first, and only exits the app
  when nothing is open. This is one of the very few Tauri commands the
  Android shell adds.

Swipe gestures (swipe-to-reply, swipe-to-mark-read) are deferred to v2.

## Identity storage: EncryptedSharedPreferences via tauri-plugin-store

The browser keeps the 32-byte Ed25519 seed in IndexedDB. On Android,
IndexedDB still exists inside the WebView but is **not** encrypted at
rest — a rooted device or backup extraction reads it plainly.

Three options:

| Option | Verdict |
|---|---|
| Keep IndexedDB (matches web) | Plain-text on disk; rejected |
| **Tauri 2's `tauri-plugin-store` on Android**, backed by Android `EncryptedSharedPreferences` (AES-256-GCM, keys in Android Keystore) | Pick this |
| Roll our own JNI bridge to Keystore | Reinvents the plugin |

The Android Keystore-backed store keeps the seed encrypted with a
hardware-bound key (StrongBox where available, TEE elsewhere). The
plugin exposes a key/value API identical in shape to the desktop's
`tauri-plugin-store`, so a single `identity-store.ts` adapter does:

```ts
// Same IdentityRecord shape as Wavvon-web (seed_hex, security_nonce, security_level)
async function loadIdentity(): Promise<IdentityRecord | null>;
async function saveIdentity(rec: IdentityRecord): Promise<void>;
```

The seed format stays byte-identical with desktop and browser. The
recovery phrase (BIP39, 24 words) remains the cross-client portability
path — a user can type their phrase on Android and end up with the same
keypair as on desktop.

**Token storage**: tokens are not identity; they go in regular Android
`SharedPreferences` (cleared on uninstall, sufficient given they can be
regenerated by signing a fresh challenge). The "remember me" /
sessionStorage distinction from the browser collapses on Android: the
device is personal, the token always persists, and the OS handles
uninstall cleanup.

## APK signing: a single self-signed release key, checked-in instructions

Side-loading only — no Play Store (store policies on federated chat are
a fight we're not picking, and the Play Console fee is a barrier).
Users download an APK from `releases.wavvon.io` and install it via
"Install from unknown sources".

| Option | Verdict |
|---|---|
| Ship debug-signed APKs | Android refuses to upgrade a debug-signed app over a release-signed one and vice-versa; flips on each CI run if the debug key isn't pinned. Awkward. |
| **One long-lived self-signed release keystore, stored as a CI secret** | Pick this |
| Per-developer keys | Breaks upgrades when a different developer cuts a release |

**Setup**:

- Generate once: `keytool -genkey -v -keystore wavvon.keystore -alias wavvon -keyalg RSA -keysize 4096 -validity 36500`
- Store base64-encoded as the CI secret `ANDROID_KEYSTORE_BASE64`.
- Password as `ANDROID_KEYSTORE_PASSWORD`; key alias password as
  `ANDROID_KEY_PASSWORD`.
- The release workflow decodes the keystore into a temp file and points
  Tauri at it via `TAURI_ANDROID_KEY_*` env vars (mirrors the Tauri
  Android signing docs).
- Treat the keystore as **permanent**: losing it means every existing
  user must uninstall and reinstall (Android refuses cross-signature
  upgrades). Back the secret up out-of-band.

The `tauri-plugin-updater` does **not** run on Android — the Android
package model is different and the plugin lacks an Android backend
today. The app polls `releases.wavvon.io/latest.json` for an Android
entry and offers a "download APK" intent; the user re-installs
manually. Crude but matches the side-loaded distribution model.

## CI plan: separate workflow, on-demand only

The current `build.yml` runs on every PR across Windows and Linux in
~3-5 minutes. Adding Android (SDK + NDK + Gradle + first build) adds
~15-20 minutes and ~3 GB of disk. PR latency is more valuable than
"Android compiles" signal on every change.

**Plan**:

- A **separate** workflow `android.yml` runs on:
  - Push to `main` (post-merge, so PR feedback stays fast),
  - Tag `v*` (release builds),
  - Manual `workflow_dispatch` (developer can run on a branch on demand).
- Not on PRs by default. A PR touching the Wavvon-android repo's
  `android/**` paths can
  opt in via a `paths:` filter — but the default stays "fast PR
  feedback".
- Uses `ubuntu-22.04` with `android-actions/setup-android@v3` for the
  SDK, the Tauri-recommended NDK version, and aggressive caching of
  `~/.gradle` and `~/.android/sdk`. First run is slow; cached runs land
  near 5 minutes.
- Output artefact: signed release APK attached to the GitHub Release
  (release workflow) or to the workflow run (post-merge / dispatch).

Since Wavvon-android is its own repo, the release workflow lives there
and produces the APK independently of Wavvon-desktop's installer
workflow. The keystore-decoding step described above is duplicated in
that workflow.

## Implementation steps

Ordered so a developer can follow without backtracking. Each step is
self-contained and leaves the tree compiling.

1. **Bootstrap the project**. `npm create vite@latest android --
   --template react-ts` inside the Wavvon-android repo. Mirror
   Wavvon-web's `package.json` deps (React 19, noble stack, `idb`,
   `bip39`).
2. **Wire Vite aliases**. Copy `web/vite.config.ts` from Wavvon-web and
   add aliases pointing at the shared component and platform sources
   from Wavvon-desktop (`desktop/src/components`, `desktop/src/types.ts`)
   and Wavvon-web (`web/src/platform`, `web/src/identity`). The
   resolution strategy (sibling checkouts, git submodules, or an npm
   workspace) is an open question — see below. Verify a
   `tsc --noEmit` passes.
3. **Add Tauri Android scaffold**. `cargo install tauri-cli --version "^2"`
   then `cargo tauri android init` inside `android/` (Wavvon-android).
   Set `identifier: com.wavvon.android` in `tauri.conf.json`.
4. **Plug in identity storage**. Add `tauri-plugin-store` (Android
   build feature). Write `platform-android/identity-store.ts` exposing
   the same `IdentityRecord` shape as Wavvon-web. Swap the web's
   `identity/store.ts` import in `App.tsx` via a build-time flag —
   simplest is a Vite `define` (`__WAVVON_PLATFORM__: 'android'`).
5. **Add the `MobileShell` component** to `desktop/src/components/`
   in Wavvon-desktop. Refactor `App.tsx` to render `MobileShell` below
   the 768 px breakpoint (or always on Android). No new logic — only
   layout. The desktop client picks this up automatically when run on
   a narrow window.
6. **Mobile CSS overlay**. `styles-mobile.css` imports the shared
   `styles.css` and applies touch-target sizing, drawer transitions,
   safe-area-inset for the status bar.
7. **OS integration plugins**. Register
   `tauri-plugin-notification`, `tauri-plugin-deep-link`,
   `tauri-plugin-dialog` for attachment picking. Wire a Rust back-press
   handler that emits an event the UI catches.
8. **Generate the signing keystore**. Document the keytool command in
   `SIGNING.md` at the root of Wavvon-android. Store the keystore
   base64 + passwords as GitHub secrets on that repo.
9. **Add `android.yml` workflow**. Triggers: `push` on main, `tag v*`,
   `workflow_dispatch`. Steps: setup JDK 17, setup Android SDK,
   restore Gradle cache, decode keystore, `cargo tauri android build
   --target aarch64 --target armv7 --release`, upload APK artefact.
10. **Manual update prompt**. On app start, fetch
    `releases.wavvon.io/latest.json`. If `platforms["android-arm64"].version`
    exceeds the installed version, show a one-time toast linking to the
    APK URL via `tauri-plugin-shell`'s `open()`.
11. **Document side-loading** in [hosting.md](hosting.md) or a new
    `docs/install-android.md`: "enable Unknown Sources, download from
    releases.wavvon.io, expect a one-time Play Protect warning."

Voice, screen share, swipe gestures, QR multi-device pairing, and Play
Store distribution are all deferred to follow-up designs.

## Open questions

| Question | Working answer |
|---|---|
| **iOS** | Out of scope. Apple Developer Program ($99/yr) + App Store review of a federated chat client is the same cost/risk wall as macOS. Revisit when the project has budget. |
| **Background WebSocket on Android** | Android aggressively kills backgrounded WebViews. v1 accepts this: messages received while backgrounded surface via the next foreground reconnect; notifications arrive only when the app is open. A future design uses a Rust foreground service via JNI to keep the WS alive — significant work, deferred. |
| **Push notifications (FCM)** | Tied to the background-WS question. FCM is the only practical Android background-delivery path, requires a hub-side push registration table, and depends on Google's push infrastructure. Designed alongside Web Push (see [browser-client.md](browser-client.md)). Deferred. |
| **Tablet / landscape layout** | The 768 px breakpoint is the only gate today; landscape phones fall above the breakpoint and get the desktop layout, which is wrong. A follow-up pass introduces a "compact desktop" mode for tablets. |
| **WebView version skew** | The Android System WebView is updated by Google Play independently of the OS. Some users on de-Googled Android pin an old WebView. v1 documents "needs Chromium-based WebView 100+". |
| **F-Droid distribution** | Possible (the repo is open-source and we use no proprietary deps), but F-Droid's reproducible-build requirements and their own signing layer add complexity. Deferred until the side-loaded APK story is stable. |
